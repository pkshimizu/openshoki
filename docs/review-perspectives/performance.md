# レビュー観点: パフォーマンス

明らかな非効率・リソースの無駄を見る。openshoki は**常駐**して動き続け、録音という
リアルタイム性のある処理を持つため、「常時の軽さ」と「録音中の取りこぼし防止」が要点。

## 見るポイント

### 常駐時のリソース消費（アイドル）
- 録音していないときに CPU を使い続けていないか（ビジーループ、無駄なポーリング、過剰なタイマー）。
- イベント駆動で待てるところをスピンウェイトしていないか。
- メモリリーク・解放漏れ（spawn したスレッド、保持したままのバッファ）がないか。常駐で蓄積する。

### 録音中のリアルタイム性
- 音声コールバック／録音ループ内で、重い処理（ファイル I/O、確保、ロック待ち、ログ整形）を
  直接行っていないか。取りこぼし・グリッチの原因になる。録音はバッファに積み、書き出しは別スレッドへ。
- バッファサイズ・サンプルレートの扱いが妥当か。過小でアンダーラン、過大でレイテンシ増。

### メモリ・割り当て
- ホットパス（音声処理ループ）での毎回の `Vec`/`String` 確保や `.clone()` を避けているか。
  バッファを使い回せないか。
- 不要な中間コレクション（`collect()` 連発）、`collect()` 後すぐ `iter()` の往復。

### I/O
- ファイル書き込みがバッファリングされているか（`BufWriter` 等）。1 サンプルごとの write 等になっていないか。
- 同期 I/O で UI／録音スレッドをブロックしていないか。

### 計測の姿勢
- 「速くするための複雑化」が、計測に基づかない早すぎる最適化になっていないか
  （可読性とのトレードオフ。ホットパス以外は読みやすさを優先）。
