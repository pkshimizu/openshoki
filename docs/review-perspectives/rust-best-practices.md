# レビュー観点: Rust ベストプラクティス

Rust らしい書き方ができているか、言語・標準ライブラリ・エコシステムの作法に沿っているかを見る。
「動くか」ではなく「Rust として素直か・将来壊れにくいか」を問う観点。

## 見るポイント

### エラーハンドリング
- `Result` / `Option` を握りつぶしていないか。`unwrap()` / `expect()` を本番経路で使っていないか
  （初期化など失敗が即終了で妥当な箇所は `expect("理由")` で根拠を残す）。
- ライブラリ層は `thiserror` 等で型付きエラー、アプリ層は `anyhow` 等で文脈付与、と層で
  使い分けられているか。`?` でエラーを適切に伝播しているか。
- エラーに文脈（どの操作で失敗したか）が付いているか。

### 所有権・借用・型
- 不要な `.clone()` がないか。借用で済むものを所有権で受け取っていないか。
- API は受け取りすぎていないか（引数は `&str` / `&[T]` / `impl AsRef<_>` 等、最小の借用で受ける）。
- ライフタイムを無理に複雑化していないか。`Rc`/`RefCell` で済むところを `unsafe` にしていないか。

### イディオム
- 手続き的なループより、イテレータアダプタ（`map`/`filter`/`collect` 等）で表現できないか。
- `match` / `if let` / `let else` を適切に使い、ネストを浅くしているか。
- newtype や `enum` で状態を型で表現できる箇所を、フラグや文字列で表していないか
  （例: 録音状態を `bool` 2 つではなく `enum RecordingState` で持つ）。

### API 設計・命名
- 公開 API に Rust API Guidelines に沿った命名（`as_`/`to_`/`into_` の使い分け、`is_`/`has_` 等）。
- `#[derive(Debug)]` など慣例的な derive が付いているか。`Default` で済む初期化を手書きしていないか。

### 並行・非同期（常駐アプリで重要）
- UI スレッドと録音スレッド間の共有が `Arc<Mutex<_>>` / channel で適切に行われているか。
- ロックを跨いで長時間保持していないか（録音ループ中に UI ロックを握り続ける等）。
- async を使う場合、ブロッキング I/O を非同期ランタイム上で直接呼んでいないか。

### 依存・ツール
- `cargo fmt` / `cargo clippy` の警告が残っていないか。`#[allow(...)]` を安易に付けていないか。
- 標準ライブラリで足りるものに重い依存を足していないか。
