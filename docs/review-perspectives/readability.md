# レビュー観点: 可読性

後から読む人（自分・他の開発者・AI エージェント）が、最小の労力で意図を理解できるかを見る。
動作の正しさではなく「読んで分かるか・直しやすいか」を問う。

## 見るポイント

### 命名
- 名前から役割が分かるか。略語・曖昧語（`data`, `tmp`, `flag`, `do_it`）を避けているか。
- `docs/CONTEXT.md` の用語（`常駐`, `録音セッション` 等）と実装の語が一致しているか。
- 真偽値・関数名が内容と一致しているか（`is_`/`has_`、副作用のある関数が問い合わせ風でないか）。

### 関数・モジュールの構造
- 1 関数が 1 つのことをしているか。長すぎる関数・深いネストを分割できないか。
- 早期 return / `let else` でガード節を使い、本筋を浅く保てているか。
- 関連する型・関数が適切なモジュールにまとまっているか。録音ロジックと UI ロジックが混在していないか。

### コメント
- 「何をしているか（コードを読めば分かること）」ではなく「なぜそうしているか」を書いているか。
- OS 固有の都合・回避策・前提など、コードからは読み取れない背景にコメントがあるか。
- コメントとコードが食い違っていないか（古いコメントの放置）。

### 一貫性
- 既存コードの書き方（エラー処理・命名・モジュール分け）と揃っているか。浮いた書き方をしていないか。
- マジックナンバー（サンプルレート・バッファサイズ等）に名前が付いているか（`const`）。

### 認知負荷
- 読み手が複数箇所を行き来しないと理解できない構造になっていないか。
- 不要な抽象化・過度な汎用化で、かえって読みにくくなっていないか（YAGNI）。
