# レビュー観点: セキュリティ

機微なデータの扱い・権限・外部入力の検証を見る。openshoki は**音声**という極めて機微な
データを扱う常駐アプリなので、プライバシー面の配慮が特に重要。

## 見るポイント

### 音声・録音データの扱い（最重要）
- 録音データ・録音ファイルが、ユーザーの意図しない場所に保存・送信されていないか。
- 録音中であることがユーザーに明示されるか（無断録音にならない設計か）。トレイアイコン等での状態提示。
- 一時ファイル・バッファに残った音声が、不要になった後も残存していないか。保存先のパーミッションは適切か。
- 外部送信（クラウド・解析）が発生する場合、その経路が TLS で保護され、必要最小限か。

### マイク権限・OS 連携
- マイクアクセス権限を適切に要求し、拒否時に安全に振る舞うか（パニックせず機能を縮退）。
- OS の録音 API / トレイ API（FFI）の戻り値を検証しているか。失敗時に未定義状態へ進まないか。

### 入力検証
- 設定値・ファイルパス・ファイル名にユーザー由来の値を使う際、パストラバーサルや不正パスを弾くか。
- 設定ファイル・外部入力のデシリアライズで、想定外の値を安全に扱うか。

### 機微情報
- ログ・エラーメッセージ・デバッグ出力に、録音内容や保存パス等の機微情報を出していないか。
- 設定に資格情報（クラウド連携トークン等）が含まれる場合、平文でログ・リポジトリに出ていないか。

### 依存・サプライチェーン
- 依存クレートに既知の脆弱性がないか（`cargo audit` 相当の観点）。
- 不要に強い権限・広いスコープのクレートを導入していないか。

### unsafe / メモリ安全
- `unsafe` ブロックが安全性を損なっていないか。FFI 境界でのバッファ長・ポインタ検証。
