# レビュー観点: Rust アンチパターン

Rust で「やってはいけない／避けるべき」典型パターンが混入していないかを見る。
ベストプラクティスの裏返しで、こちらは「具体的な地雷の検出」に集中する。

## 検出したいアンチパターン

### パニック・握りつぶし
- 本番経路の `unwrap()` / `expect()` / `panic!()` / `unreachable!()`。特にユーザー入力・I/O・
  デバイス（マイク）状態に依存する箇所。
- 配列の添字アクセス `v[i]`（範囲外パニック）。`get(i)` で回避できないか。
- `let _ = result;` でエラーを捨てている、`Result` に `#[must_use]` 無視の放置。

### unsafe の濫用
- 安全な代替があるのに `unsafe` を使っている。`unsafe` ブロックに「なぜ安全か」のコメントが無い。
- FFI（OS の録音／トレイ API 呼び出し）の戻り値・ポインタを検証せず使っている。

### クローン地獄・非効率
- 借用で済むのに `.clone()` / `.to_string()` / `.to_vec()` を連発（特にホットパスや音声バッファ）。
- `collect()` してすぐ `iter()` し直すなど、無駄な中間コレクション。
- `String` を引数に取って中で `&str` 化するだけ（`&str` で受けるべき）。

### 型・状態の表現ミス
- 「ありえない状態」を型で禁止せず、`bool` の組み合わせや `Option` のネストで表現している
  （例: `is_recording: bool` と `is_paused: bool` の同時 true が無意味なら enum にする）。
- stringly-typed: enum で表せる状態を文字列比較で分岐している。

### 並行性の罠
- `Mutex` のロックを保持したまま `.await` / 別ロック取得 → デッドロックの温床。
- `Arc<Mutex<_>>` を雑に共有し、ロック粒度が大きすぎる。
- スレッドを spawn しっぱなしで join / 終了管理していない（常駐アプリで終了時にハングの原因）。

### その他
- `as` による無検証の数値キャスト（切り捨て・符号誤り）。`try_into()` を検討。
- `clippy` の警告を `#[allow(...)]` で黙らせているだけで根本対処していない。
- 巨大な `match` / 深いネスト / 1 関数に責務過多。
