# メニューからの録音の開始・停止（マイク音声）

- 作成日: 2026-06-28
- ステータス: ドラフト

## 概要

トレイメニューから **録音セッション** の開始・停止を操作できるようにする。本プランでは
まず **マイク音声** を取得し、停止までを 1 つの MP3 ファイルとして保存先に書き出す。
将来の文字起こしを見据え、音源ごとに別ファイルへ保存する方針とし、その「音源を 1 つ持つ
録音基盤」をここで作る（スピーカー等のシステム音声取得は別プラン／別 issue）。openshoki の
中核「常駐したまま素早く録音を開始・停止する」体験を実現する最初の機能。

## 背景・前提（コンテキスト）

- openshoki は **常駐**型の録音アプリ。GUI は Slint、トレイ常駐は tray-icon。トレイの
  メニューイベントを `slint::Timer` でポーリングし、Slint イベントループ（メインスレッド）上で
  処理する（`src/main.rs` の `build_menu_event_handler`、`docs/CONTEXT.md`）。
- 既存メニューは「ウィンドウを表示/隠す」トグル（`toggle_item`）と「終了」（`quit_item`）の
  2 項目（`src/tray.rs`）。トグルは状態に応じてラベルを切り替え、ラベル文字列は定数に
  一本化する方針（`docs/rules/coding-conventions.md`）。
- **録音セッション** は録音開始から停止までの 1 回分。本プランからは、1 回の録音セッションが
  音源ごとに分かれた複数ファイルを生むことがある（将来の文字起こしのため音源を混ぜない）。
- 録音は `cpal`（マイク取得）+ `mp3lame-encoder`（libmp3lame）で `.mp3` 出力、という方針が
  確定済み（`docs/CONTEXT.md`）。
- 保存先は「録音ファイルの保存先」設定（issue #3）の `recording_dir` を使う。本プランは
  **issue #3 に依存**する（#3 を先に実装し、その `Config` を参照する）。
- エラーは握りつぶさず、`?` を使えないコールバックでは `eprintln!` でログする方針
  （`docs/rules/error-handling.md`）。

## 要件

- トレイメニューの単一トグル項目で、録音の開始と停止を切り替えられる
  （「録音を開始」⇄「録音を停止」）。
- 開始すると既定の入力デバイスから **マイク音声** を取得し、停止までを 1 つの MP3 ファイルと
  して保存先に書き出す。
- 将来の文字起こしのため、音源を混ぜず音源ごとに別ファイルへ保存する。本プランの音源は
  マイクのみだが、ファイル名に音源が分かる接尾辞を付け、後でシステム音声ファイルを並べて
  置けるようにする（例: `openshoki-YYYYMMDD-HHMMSS-mic.mp3`）。
- 保存先は issue #3 の設定（`Config.recording_dir`）を参照する。保存先が無ければ作成する。
- 録音中であることを、メニューラベルの切り替えに加えてトレイの表示
  （ツールチップ／アイコン）でも示す。
- デバイス取得失敗・保存失敗などは握りつぶさずログし、アプリ（常駐）は落とさない。
- スコープ外:
  - **スピーカー等のシステム音声の取得**（別プラン `docs/plans/20260628-system-audio-capture.md`、
    macOS の ScreenCaptureKit 先行）。
  - マイクとシステム音声のミキシング（混ぜずに別ファイルにするため不要）。
  - 入力デバイスの選択 UI（既定デバイスのみ）。一時停止／再開、音量メーター、自動分割。
  - ファイル形式・ビットレート等の設定 UI（実装時の既定値で固定）。

## 確定した論点

ユーザー確認で決定:

1. **録音スタック**: `cpal` で既定入力デバイスから PCM を取得し、`mp3lame-encoder` で MP3 に
   エンコードして書き出す。形式は `.mp3`（互換性重視）。
2. **メニュー UI**: 単一トグル項目。録音状態は別フラグを持たず、保持している録音セッション
   （`Option<Recorder>`）の有無から判断する（「ありえない状態」を作らない方針）。
3. **保存先**: issue #3 の `Config.recording_dir` を使う。本プランは #3 に依存し、#3 完了後に着手。
4. **録音中の状態表示**: メニューラベルの切り替えに加え、録音中はトレイのツールチップ
   （例: 「録音中…」）とアイコンを変える。
5. **音源ごとに別ファイル**: 文字起こしを見据え、音源を混ぜず別ファイルに保存する。本プランは
   マイクのみだが、音源接尾辞（`-mic`）を付けてシステム音声ファイル（`-system`）を後で並べられる
   ようにする。
6. **マイク先行・システム音声は別 issue**: スピーカー等のシステム音声取得は OS 依存で難度が
   高いため分割し、本プランはマイク＋別ファイル出力の土台までを担う。

調査で解消した点:
- 既存メニュー構造（toggle/quit）とイベント処理経路（`build_menu_event_handler`）を確認。
  録音トグルは同じ仕組みに項目を 1 つ足す形で載せられる。
- issue #3（保存先設定）は未実装（open）。よって本プランは #3 に依存する。
- クレート最新版: `cpal 0.18`、`mp3lame-encoder 0.2`、タイムスタンプ命名に `chrono 0.4`。

## 実装方針

- **録音を担う `src/recorder.rs` を新設**する。トレイ／UI から独立した「録音セッションの
  開始・停止とファイル書き出し」モジュールとして責務を分ける。
  - `Recorder` は実行中の録音セッションを表す。`Recorder::start(recording_dir: &Path)
    -> Result<Recorder, ...>` で開始、`Recorder::stop(self) -> Result<Vec<PathBuf>, ...>` で
    停止してエンコーダを flush し、ファイルを確定して保存パス（音源ごと）を返す。
  - 音源は「独立したエンコーダ＋出力ファイルを持つキャプチャ」を 1 単位として扱えるように
    設計する。本プランの音源はマイク 1 つだが、停止/flush の流れを音源単位にしておき、後続の
    システム音声を 2 つ目の音源として無理なく足せるようにする（過度な抽象化はしない）。
  - 開始時（マイク）: `cpal` の既定ホスト／既定入力デバイスと既定入力設定を取得し、入力
    ストリームを構築。`mp3lame-encoder` をサンプルレート・チャンネル数・ビットレートで初期化し、
    保存先にタイムスタンプ＋`-mic` 名の `.mp3` を作成（`create_dir_all` で保存先を用意）。
    エンコーダと出力先を `Arc<Mutex<...>>` で共有してコールバックへ渡し、`stream.play()` する。
  - 入力サンプル形式（f32／i16 等）はデバイス既定に従い、LAME が受け取れる形式へ変換する。
  - 停止時: ストリームを drop してコールバックを止めてから、エンコーダを flush して末尾の
    フレームを書き、ファイルを閉じる（録音末尾を取りこぼさない順序にする）。
  - `cpal::Stream` は `Send` でない場合があるため、メインスレッド（Slint イベントループ）上で
    のみ保持し、スレッドを跨いで移動させない。
- **`src/tray.rs` に録音トグル項目を追加**する。
  - `record_toggle_item: MenuItem` を追加し、ラベルは `RECORD_LABEL_START`（「録音を開始」）/
    `RECORD_LABEL_STOP`（「録音を停止」）の定数に一本化。
  - 録音中・待機中でトレイのツールチップ／アイコンを切り替える手段を用意する。
- **`src/main.rs` で配線**する。
  - 録音セッションを `Option<Recorder>` として保持し、録音トグルのメニューイベントで開始/停止を
    切り替える。`None`→開始（`Config.recording_dir` を渡す）→`Some` にしてラベル「録音を停止」、
    トレイ表示を録音中へ。`Some`→`take()` して `stop()`→`None` にしてラベル「録音を開始」、
    トレイ表示を待機中へ戻す。
  - 開始に必要な `recording_dir` は issue #3 の `Config`（起動時 `load()` 済み）から取得する。
  - 失敗は `eprintln!` でログし、状態は変更せず待機中のままにする。`Recorder` は `!Send` の
    `Stream` を持つため、ハンドラ（メインスレッドのクロージャ）内で所有する。
- 既存のウィンドウ表示/非表示・終了・常駐（`run_event_loop_until_quit`）の挙動は変更しない。

## 実装ステップ

1. **依存追加**: `cargo add cpal mp3lame-encoder chrono`。libmp3lame のビルド（C コンパイラ）を
   含め `cargo build` が通ることを確認する。
2. **`src/recorder.rs` を実装**: `Recorder::start/stop`、既定デバイス取得、ストリーム構築、
   サンプル形式変換、LAME エンコード、タイムスタンプ＋`-mic` 名でのファイル書き出しと flush。
   `main.rs` に `mod recorder;` を追加。`cargo build`・`cargo clippy` が通ることを確認。
3. **録音の単体動作確認**: `examples/` の確認用バイナリ等で、数秒 `start`→`stop` し、保存先に
   `-mic.mp3` が生成され、再生して音声が入っていることを確認（確認後、一時バイナリは削除）。
4. **`src/tray.rs` に録音トグル項目を追加**: `record_toggle_item` と `RECORD_LABEL_START/STOP`、
   録音中/待機中の表示切り替え。`cargo build` を確認。
5. **`src/main.rs` を配線**: `Option<Recorder>` の保持、録音トグルでの開始/停止、ラベルと
   トレイ表示の更新、`Config.recording_dir` の参照、失敗時のログ。
6. **検証**: `cargo build` / `cargo fmt --check` / `cargo clippy --all-targets -- -D warnings`。
   目視で、トレイの「録音を開始」→ 録音中表示 →「録音を停止」→ 保存先に再生可能な `-mic.mp3` が
   生成される、ことを確認する。

## 影響範囲・リスク

- 影響を受けるファイル/モジュール:
  - 新規: `src/recorder.rs`
  - 変更: `Cargo.toml`（依存追加）、`src/main.rs`（`mod recorder`・`Option<Recorder>` 保持・
    録音トグル配線）、`src/tray.rs`（録音トグル項目・状態表示）
  - 依存: issue #3（`Config.recording_dir`）。#3 完了後に着手する。
- リスクと対策:
  - **macOS のマイク権限（TCC）**: `cargo run` の素のバイナリは `Info.plist` の
    `NSMicrophoneUsageDescription` が無いと権限プロンプトが出ず、録音が無音／失敗することが
    ある。配布用バンドル化・権限記述はパッケージングの段階で扱う。当面は権限を付与した環境で
    目視検証し、失敗時はログで原因を残す。
  - **`cpal::Stream` が `!Send`**: メインスレッド上でのみ保持し、スレッドを跨がない。
  - **サンプル形式・レートの差異**: デバイス既定の形式とレートに合わせて変換し、LAME を対応
    レートで初期化する。リサンプリングは行わずデバイスレートを使う。
  - **リアルタイムコールバック内エンコード**: 音声コールバック内で `Mutex` ロックと MP3 エンコードを
    行うと音飛びの懸念がある。まず単純実装で進め、グリッチが出たらコールバックはチャネルへ送る
    だけにし、エンコード／書き出しを専用スレッドへ分離する。
  - **libmp3lame のビルド依存**: ピュア Rust 構成から外れ、C ライブラリのビルドが必要。各 OS で
    ビルドが通るかをステップ 1 で確認する。
  - **停止時の取りこぼし**: ストリーム停止 → flush → ファイル確定の順序を守る。

- 録音中／待機中のトレイアイコンの具体デザイン（暫定アイコンの差し替えは後続でも可）。
