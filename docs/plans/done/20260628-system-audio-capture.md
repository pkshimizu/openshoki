# スピーカー出力（システム音声）の録音（macOS 先行）

- 作成日: 2026-06-28
- ステータス: ドラフト

## 概要

録音セッション中に、マイクだけでなく **スピーカー等から出力されているシステム音声**
（再生中の音）も取得し、マイク音声とは **別ファイル** の MP3 として保存する。将来の文字起こしで
発話（マイク）と相手側／再生音（システム）を分けて扱えるようにするのが狙い。macOS を先行
対象とし、ScreenCaptureKit を用いる。

## 背景・前提（コンテキスト）

- openshoki は **常駐**型の録音アプリ（macOS 優先、`docs/CONTEXT.md`）。録音はトレイの単一
  トグルで開始・停止する（マイク録音プラン `docs/plans/20260628-recording-start-stop.md`）。
- マイク録音プランで `src/recorder.rs` を新設し、録音セッションを `Option<Recorder>` として
  メインスレッドで保持、音源ごとに独立したエンコーダ＋出力ファイルを持つ設計にしてある。
  本プランは **その録音基盤に 2 つ目の音源（システム音声）を追加**する形を取る。
- 録音は MP3 出力（`mp3lame-encoder`）で統一する（`docs/CONTEXT.md`）。システム音声も同じ
  エンコード経路を再利用し、ファイルは別にする。
- 保存先は issue #3 の `Config.recording_dir`。本プランは **issue #3** と **マイク録音 issue** の
  両方に依存する（録音基盤が先）。
- システム音声のループバック取得は OS 依存で、共通 API が無い。macOS は ScreenCaptureKit、
  Windows は WASAPI loopback、Linux は monitor source、と方式が分かれる。

## 要件

- 録音セッションの開始・停止に連動して、システム音声を取得し、マイクとは別の MP3 ファイルに
  保存する（例: `openshoki-YYYYMMDD-HHMMSS-system.mp3`）。同一セッションのマイクファイルと
  タイムスタンプを揃え、ペアだと分かるようにする。
- マイクとシステムは混ぜず、それぞれ独立した MP3 にする（文字起こしのため）。
- 開始/停止は既存の単一トグルのまま。ユーザー操作は増やさない（1 操作でマイクとシステムの
  両方が始まり、両方が止まる）。
- システム音声の取得・保存に失敗しても、マイク録音やアプリ常駐は巻き込まず、握りつぶさず
  ログする。権限拒否時は分かるようにする。
- スコープ外:
  - Windows（WASAPI loopback）/ Linux（monitor source）のシステム音声取得（後続プラン）。
  - アプリ別・デバイス別の出力選択（システム全体の出力を対象とする）。
  - マイクとシステムのミキシング、話者分離、文字起こしそのもの。

## 確定した論点

ユーザー確認で決定:

1. **macOS の取得方式**: ScreenCaptureKit を使う（`screencapturekit` クレート）。macOS 13+ で
   動作し、仮想オーディオドライバのインストールが不要。
2. **対象 OS の範囲**: macOS を先行実装し、Windows / Linux は後続プランに分ける。
3. **別 issue として分割**: マイク録音とは別 issue にし、録音基盤（マイク録音）完了後に着手する。
4. **別ファイル保存**: システム音声はマイクと混ぜず、`-system` 接尾辞の別 MP3 に保存する。

調査で解消した点:
- 実行環境は macOS 26.5。ScreenCaptureKit は十分に利用可能。
- クレート: `screencapturekit 8.0`（ScreenCaptureKit の安全な Rust バインディング、画面と音声の
  取得に対応）。MP3 エンコードは既存方針の `mp3lame-encoder` を再利用。

## 実装方針

- **`src/recorder.rs` にシステム音声の音源を追加**する。マイク録音プランで用意した「音源単位で
  エンコーダ＋出力ファイルを持つ」構造に、ScreenCaptureKit ベースの 2 つ目の音源を足す。
  - macOS 限定コードは `#[cfg(target_os = "macos")]` で隔離する（Dock 非表示の既存 `hide_dock_icon`
    と同じ方針）。他 OS では当面システム音源を持たない（マイクのみ）ビルドになる。
  - 開始時: ScreenCaptureKit で **音声のみ** のキャプチャを構成し（`SCStreamConfiguration` で
    音声を有効化、画面は最小構成）、出力されるオーディオサンプルバッファを受け取るハンドラを
    登録する。サンプルを LAME へ渡し、`-system` 名の `.mp3` に書き出す。
  - 停止時: キャプチャを停止してから、システム音源のエンコーダを flush してファイルを確定する。
    マイク音源と同じ「停止 → flush → 確定」の順序に揃える。
  - フォーマット（サンプルレート・チャンネル数）は ScreenCaptureKit の音声出力に合わせ、必要なら
    LAME 側をそのレートで初期化する。
- **開始/停止の連動**: `Recorder::start` がマイクとシステムの両音源を起動し、`stop` が両方を
  停止・確定して、生成した全ファイルのパスを返す。トレイのトグルや `main.rs` の配線は
  マイク録音プランのまま（音源が増えても操作は不変）。
- **権限（TCC）**: ScreenCaptureKit は画面収録の権限を要求する。初回は OS のプロンプト／設定
  誘導が要る。権限が無い場合は取得を諦め、ログを残してマイク録音は継続する。

## 実装ステップ

1. **前提確認**: マイク録音 issue と issue #3 が完了し、`src/recorder.rs` に音源単位の
   開始/停止/flush の枠組みがあることを確認する。
2. **依存追加**: `cargo add screencapturekit`（macOS ターゲット限定の依存として追加）。
   `cargo build` が macOS で通ることを確認。
3. **システム音声キャプチャの単体確認**: `examples/` の確認用バイナリ等で、ScreenCaptureKit の
   音声のみキャプチャを数秒動かし、サンプルが届くこと・`-system.mp3` が生成され再生して
   システム音声が入っていることを確認する（確認後、一時バイナリは削除）。権限プロンプトの
   挙動もここで確認する。
4. **`src/recorder.rs` に音源を統合**: マイクと並ぶ 2 つ目の音源として組み込み、`start`/`stop` が
   両音源を扱うようにする。`#[cfg(target_os = "macos")]` で隔離。`cargo build`・`cargo clippy` を確認。
5. **検証**: `cargo build` / `cargo fmt --check` / `cargo clippy --all-targets -- -D warnings`。
   目視で、トレイの「録音を開始」→ 何か音を再生 →「録音を停止」→ 保存先に `-mic.mp3` と
   `-system.mp3` がペアで生成され、システム側に再生音が入っていることを確認する。

## 影響範囲・リスク

- 影響を受けるファイル/モジュール:
  - 変更: `src/recorder.rs`（システム音声の音源追加）、`Cargo.toml`（macOS 限定で
    `screencapturekit` 追加）。トレイ／`main.rs` の操作系は原則変更なし。
  - 依存: issue #3（保存先）と マイク録音 issue（録音基盤）。両方の完了後に着手する。
- リスクと対策:
  - **画面収録権限（TCC）**: ScreenCaptureKit は画面収録権限を要する。`cargo run` の素のバイナリ
    では権限付与・プロンプト挙動が不安定なことがある。配布用バンドル化・権限記述はパッケージング
    段階で扱う。権限が無ければシステム音源は諦めてログし、マイク録音は止めない。
  - **ScreenCaptureKit が音声と一緒に画面取得を要求する設計**: 音声のみを最小コストで取れる構成を
    探す（ダミーの最小ディスプレイ構成など）。CPU 負荷・取得遅延を単体確認で見る。
  - **`screencapturekit` クレートの API 安定性／バインディングの過不足**: 必要な音声 API が無い／
    薄い場合は、`cidre` や CoreAudio process tap への切り替えを検討する（単体確認の段階で判断）。
  - **マイクとの同期**: 厳密なサンプル単位同期は狙わず、同一セッションで同時に開始・停止し、
    タイムスタンプを揃える程度にする（文字起こし用途には十分）。
