# アプリのベース作成（Rust 基盤＋トレイ常駐＋Slint）

- 作成日: 2026-06-27
- ステータス: ドラフト

## 概要

openshoki の土台となる Rust アプリを立ち上げる。メニューバー／タスクバーに**常駐**し、
トレイメニューから Slint 製ウィンドウを表示・非表示でき、クリーンに終了できる骨組みを作る。
録音機能はこの段階では実装せず、後続の issue に回す。これは「常駐型録音アプリ」の他の
すべての機能が乗る基盤を、最初に検証可能な形で固めるためのもの。

## 背景・前提（コンテキスト）

- `docs/CONTEXT.md` のとおり、openshoki はウィンドウを主役にせず**常駐**して素早く録音を
  操作する体験を狙う。よって起動時はウィンドウを出さず、トレイのみで存在する設計が土台になる。
- 言語は Rust。GUI には **Slint** を使う（要件で確定）。トレイ常駐は標準ライブラリでは
  賄えないため別クレートを使う。
- リポジトリは初期状態（`.git` のみ）。既存コードとの整合という制約はなく、ゼロから構成を作る。

## 要件

- Rust のアプリ基盤（`Cargo.toml`・ディレクトリ構成・ビルド設定）を整備する。
- メニューバー（macOS）／タスクバー（Windows・Linux）に常駐する。
- GUI は Slint で実装する。
- 起動時はウィンドウを表示せず、トレイに常駐する。トレイメニューからウィンドウを開閉できる。
- トレイメニューから終了できる。
- スコープ外:
  - 録音機能（マイク取得・録音セッション・ファイル保存）。別 issue で実装する。
  - 設定の永続化、自動起動、アップデート、パッケージング（`.app`/インストーラ生成）。
  - Windows・Linux での作り込み・検証（基盤は載せるが、まず macOS で動作確認する）。

## 確定した論点

ユーザー確認および設計判断で次を確定した。

- **スコープは骨組みのみ**（ユーザー確認）。トレイ常駐・ウィンドウ表示/非表示・終了まで。
  録音は含めない。
- **macOS 優先**（ユーザー確認）。クレートはクロス対応のものを選ぶが、動作確認はまず macOS で行う。
- **起動時はウィンドウを隠しトレイ常駐**（ユーザー確認）。メニュー項目から開く。
- **トレイクレートは `tray-icon` を採用**（設計判断）。Slint と同じく winit ベースの
  エコシステムで動き、macOS/Windows/Linux に対応。メニューは `tray-icon` が内部利用する
  `muda` 経由で構築する。Slint 単体にはトレイ常駐 API が無いため、これを補う。
- **Slint と tray-icon のイベントループ統合方針**（設計判断）: Slint の既定バックエンド
  （winit ベース）のイベントループを `run()` で回し、`tray-icon` のメニュー/アイコンイベントは
  `MenuEvent::receiver()` / `TrayIconEvent::receiver()` のチャネルから受け取る。これを
  `slint::Timer`（周期ポーリング、例: 100ms）で拾い、`slint::invoke_from_event_loop` 経由で
  ウィンドウの表示/非表示・終了を行う。これは追加のイベントループ制御を要さず最小構成で済む。
  - 代替案: `i-slint-backend-winit` で winit のイベントループを直接握り、`EventLoopProxy` と
    `tray-icon` の `set_event_handler` でイベント駆動にする方法。ポーリング遅延が問題になった
    場合の将来の改善として保留（骨組みでは過剰）。
- **macOS のドックアイコン非表示**（設計判断）: 常駐アプリとして Dock にアイコンを出さない
  ため、起動時に NSApplication の activation policy を Accessory 相当にする。`cargo run` 段階では
  実行時設定で対応し、最終的な `LSUIElement` 指定はパッケージング時（スコープ外）に委ねる。

## 実装方針

`tray-icon`（常駐とメニュー）＋ Slint（ウィンドウ UI）を、Slint のイベントループ上で
協調させる構成を取る。トレイのメニューイベントをポーリングで Slint 側に橋渡しし、
ウィンドウの開閉・終了を制御する。状態（ウィンドウの表示状態など）は最小限に留める。

想定するプロジェクト構成:

```
openshoki/
├── Cargo.toml          # 依存: slint / tray-icon、ビルド依存: slint-build
├── build.rs            # slint_build::compile で .slint をコンパイル
├── ui/
│   └── app-window.slint  # ウィンドウ UI（最小の枠。後で設定 UI 等を載せる土台）
├── src/
│   ├── main.rs         # エントリ。ドック非表示→トレイ初期化→Slint イベントループ起動
│   └── tray.rs         # トレイアイコン/メニュー構築と、イベントのディスパッチ
└── assets/
    └── tray-icon.png   # トレイ表示用アイコン
```

## 実装ステップ

各ステップは「何を確認できたら完了か」を添える。

1. **Cargo プロジェクトを作る**
   `cargo init`（バイナリクレート）。`Cargo.toml` に `slint`、`tray-icon`、ビルド依存
   `slint-build` を追加。`build.rs` で `slint_build::compile("ui/app-window.slint")` を呼ぶ。
   完了条件: 空の `main` で `cargo build` が通る。
2. **Slint のウィンドウを用意する**
   `ui/app-window.slint` に最小のウィンドウ（タイトル＋プレースホルダ文言）を定義し、
   `main.rs` から生成できるようにする。まだ常時表示はしない。
   完了条件: 一時的に `.show()`→`run()` するとウィンドウが出ることを確認できる（確認後コメントアウト/削除）。
3. **トレイアイコンとメニューを表示する**
   `src/tray.rs` で `tray-icon` を使い、`assets/tray-icon.png` を読み込んでトレイアイコンを
   生成。メニュー項目「ウィンドウを表示 / 終了」を作る（録音系はプレースホルダも置かない）。
   `main.rs` でトレイを初期化し、起動時はウィンドウを出さずイベントループだけ回す。
   完了条件: macOS メニューバーにアイコンが出て、クリックでメニューが開く。
4. **メニューイベントを Slint 側へ橋渡しする**
   `slint::Timer` で `MenuEvent::receiver()` をポーリングし、項目に応じて
   `invoke_from_event_loop` でウィンドウの `show()` / `hide()`、終了処理を呼ぶ。
   ウィンドウの「表示中かどうか」を保持し、表示中はメニューを「ウィンドウを隠す」に切り替える
   （または両項目を用意）。
   完了条件: 「ウィンドウを表示」でウィンドウが開き、再操作で隠れる。「終了」でアプリが終了する。
5. **macOS でドックアイコンを隠し、起動時トレイ常駐を確定する**
   起動時に activation policy を Accessory 相当に設定し、Dock にアイコンを出さない。
   起動直後はウィンドウ非表示・トレイのみであることを確認する。
   完了条件: `cargo run` で Dock にアイコンが出ず、メニューバーに常駐し、ウィンドウは隠れている。
6. **クリーン終了と後片付けを確認する**
   終了時にトレイアイコンが残らず、プロセスが正常終了することを確認する。
   完了条件: 「終了」後にメニューバーからアイコンが消え、プロセスが残らない。

## 影響範囲・リスク

- 影響を受けるモジュール/データ: 新規作成のため既存への影響なし。`Cargo.toml`・`build.rs`・
  `src/`・`ui/`・`assets/` を新設。
- リスクと対策:
  - **Slint と tray-icon のイベントループ競合**: 両者とも winit 系。トレイは macOS では
    メインスレッド＆イベントループ稼働が前提のため、トレイ生成はイベントループ起動後
    （または起動直前のメインスレッド）に行う。問題が出たら代替案（winit バックエンド直制御）へ。
  - **macOS のドック非表示が `cargo run` で効きにくい場合**: 実行時の activation policy 設定で
    対応。最終的には `LSUIElement` 指定（パッケージング時）が確実だが、本スコープ外。
  - **トレイアイコンの見た目（macOS のテンプレート画像）**: ダーク/ライトでの視認性は
    アイコン素材の問題。骨組みでは暫定アイコンで進め、素材調整は後続に回す。
  - **ポーリング方式のオーバーヘッド**: 100ms 周期程度ならアイドル負荷は軽微。問題になれば
    イベント駆動へ切り替える（`performance.md` 観点の「アイドル時 CPU」に留意）。

## 未確定事項

- トレイメニューの最終的な項目構成（録音開始/停止・設定 等）は、録音機能の issue で確定する。
- パッケージング方法（`cargo-bundle` / `cargo-packager` 等）と `Info.plist`（`LSUIElement`）の
  扱いは、配布を検討する段階で別途決める。
- Windows・Linux での常駐挙動・アイコン要件の詰めは、クロス対応に着手する段階で行う。
